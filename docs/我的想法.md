# 我对这个博客的想法

> 写于项目初期。大部分想法已落地：静态+COS、列表+详情架构、暖白配色、衬线标题、默认可见的动画策略。暗色模式也已实现。

## 架构

### 静态优先不是"妥协"，是"最佳实践"

很多人的第一反应是"博客嘛，先做个静态的凑合，以后有钱了再上动态"。这个逻辑是反的。

静态博客不是丐版，是**最优解**。原因：

1. 博客 99% 的时间在被人读，不是在被人写。读这个动作，不需要服务器。一个 HTML 文件从 CDN 边缘节点推出去，比任何 SSR 都快。
2. 服务器会挂、会欠费、会被扫端口、要升级依赖。静态文件不会。只要你没删 COS 桶，博客就永远在线。
3. 你写到 100 篇文章的时候，构建时间可能从 5 秒变成 15 秒——这跟你没关系，GitHub Actions 在跑。

Phase 2 加 Lighthouse API 也不是"升级"，是**补缺**——只为评论、统计这种天然有状态的功能加一个最小化的后端。这不是架构演进，是功能补丁。

### 架构上我只有两个坚持

**第一，前端代码不感知后端是否存在。**

```ts
const API_BASE = process.env.NEXT_PUBLIC_API_URL || ''
```

这个设计很重要。它意味着：
- 你可以在 Phase 1 完全不碰服务器代码
- 三个月后想加评论了，Lighthouse 上部署一个 Hono 服务，改一行环境变量
- 前端不会因为 API 挂了而白屏——`if (!API_BASE) return []`

这也是为什么我不推荐直接用 Next.js SSR 模式。SSR 把前端和后端绑死在一个进程里，你没法把静态部分放 COS、动态部分放 Lighthouse。

**第二，内容归属权在你手上。**

Giscus 很好，但它本质上是把你的评论存在 GitHub Discussions 里。审核、通知、导出都受限。Phase 1 用 Giscus 可以，但 Phase 2 一定要把评论收到自己的 SQLite 里。这是你自己的博客，不是 GitHub 的分站。

## 视觉设计

### 博客的 UI 只有一个指标：连续阅读 10 分钟累不累

花哨的入场动画、粒子背景、视差滚动——这些在作品集网站上很酷，在博客里是噪音。

一个人打开你的文章，要做的事是**从第一个字读到最后一个字**。你要帮他把这件事做完，而不是打断他。

这意味着：

**行宽必须限制。** 680-720px，不能再宽。人眼从一行末尾扫回下一行开头是有物理成本的，行越宽成本越高。为什么报纸分栏、Kindle 只有巴掌宽——不是在省纸，是眼球追踪的生理限制。

**行距要大于默认值。** CSS 默认 `line-height: 1.2`，中文至少 `1.7`，推荐 `1.8`。中文字形是方块的，笔画密度比拉丁字母高，需要更大的行间距来呼吸。

**不要用纯白底色。** `#ffffff` 在屏幕上发光，长时间阅读像盯着灯泡。`#fafaf8` 暖白——或者暗色模式下的 `#111110` 暖黑——阅读体验质的区别。每个认真做阅读的产品都懂这个（Kindle、Apple Books、纽约时报、Medium），但没有一个会写在 blog 教程里。

**标题需要喘气。** `h2` 的 `margin-top` 应该明显大于 `margin-bottom`。标题的视觉作用是把文章分成段落，它应该靠近下面那段、远离上面那段。这个细节很少有人注意到，但做到了，文章呼吸感立刻不一样。

### 我只推荐一种强调色

Material Design 那种五颜六色的方案是为 SaaS 产品做的——主色、辅助色、错误色、警告色、信息色，每个按钮一个颜色。

博客不是 SaaS。你只需要一个颜色：链接的颜色。

这个颜色出现的地方：正文链接、导航当前页高亮、TOC 高亮、代码块中的字符串（如果你不调主题）、Quote 左边框。

选一个你自己看着舒服的颜色就行。蓝 `#2563eb`、绿 `#059669`、紫 `#7c3aed`，都可以。但只能有一个。

### 关于字体

标题用衬线、正文用无衬线——这个选择纯粹是个人偏好。但有一个事实是：我们日常看到的 UI 文本几乎全是无衬线（系统界面、APP、网页）。所以当你的博客标题突然出现衬线字体时，读者的大脑会收到一个信号：这里跟别的地方不一样，这是"阅读模式"。

这就是为什么 Medium 用衬线标题、为什么 Apple Books 正文用衬线。不是衬线更好看，是它能**切换心理模式**。

如果你想走这条路，`DM Serif Display` 是最省心的选择——Google Fonts 上免费，字形干净，中文配 `Noto Serif SC` 风格接近。不想折腾的话，全用 Inter + Noto Sans SC 也完全没问题。

### 暗色模式

不要用 `prefers-color-scheme` 强制跟随系统。尊重系统偏好是好事，但你得给用户一个手动切换的按钮。最好的做法：

1. 首次加载跟随系统
2. localStorage 记住用户选择
3. 用户点了切换后，以 localStorage 为准，忽略系统
4. 过渡动画 200ms，`transition: background-color 200ms, color 200ms`

这是苹果 Human Interface Guidelines 和 Google Material Design 共同推荐的方案。

### 我觉得不值得花时间的事

- **首页"酷炫"动画**。读者来博客是看文章的，不是看动画的。卡片 hover 上移 2px 够用了。
- **完美像素的 Figma 设计稿**。个人博客不是产品交付，Tailwind 的类名就是设计语言。把 `padding: 1.5rem` 改成 `p-4` 还是 `p-6`，在页面上看一眼就知道，不需要提前设计。
- **侧边栏"热门文章""标签云"**。没人点。读者要么从搜索引擎进来直接看文章，要么从首页按时间翻。标签云是 2005 年的产物。
- **Facebook/Twitter 分享按钮**。大多数读者的浏览器都自带分享功能。多放几个第三方按钮只会让你的页面多加载几百 KB 的 JS。

### 如果只让我挑三个视觉细节投入时间

1. **排版**：680px 行宽 + 1.8 行距 + 暖白底 + h2 上下间距不对称
2. **代码块**：shiki 高亮 + 行号 + 暗色模式自动切换——技术博客的代码块就是第二门面
3. **字体加载**：`next/font` 自动优化，不要让字体加载导致布局抖动

这三个做对了，读者读你文章时不会注意到"设计"，只会觉得"舒服"。做到了就是最好的设计。

---

## 总结

架构上：COS 存前端 + Lighthouse 跑 API + `API_BASE` 解耦。够用、省钱、不折腾。

设计上：把读者的 10 分钟阅读体验放在第一位。排版 > 配色 > 动效。克制比炫技难，但也更有价值。

你的博客不需要让人"哇"，需要让人读完。
