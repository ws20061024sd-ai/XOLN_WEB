## 一、待办：COS 上传增量优化

### 问题

每次上传 2177 个文件约 4-7 分钟，但通常只改了几篇 `.md` 内容。全量上传浪费大量时间。

### 方案

**原理**：上传脚本计算每个文件的 MD5 哈希，存到 `cos-manifest.json`。下次上传时对比，只传改动过的文件。

**具体做法**：

修改 `upload.js`：
- 遍历 `out/` 下所有文件，计算 MD5（或 SHA256）
- 读取上次的 `cos-manifest.json`（存在 COS 桶根目录）
- 对比差异：
  - 哈希不同 → 上传
  - 新文件 → 上传
  - 哈希相同 → 跳过
  - 旧 manifest 有但 out/ 没有 → 从 COS 删除
- 上传完后更新 manifest，顺便把它也上传到 COS

**效果预估**：

| 场景 | 改动量 | 上传时间 |
|------|--------|---------|
| 只改了一篇 .md 内容 | ~10 个文件 | < 5 秒 |
| 改了组件/代码 | ~100 个文件 | < 1 分钟 |
| 首次上传 | 全量 2177 | 4-7 分钟（不变） |
| `rm -rf .next && rebuild` | 全量（chunk hash 全变） | 4-7 分钟 |

**注意事项**：
- `.next` 缓存保留时，`_next/static/chunks/` 下的 JS 文件名含内容哈希，改代码才会变
- 只改 .md 时，仅受影响的 HTML 页面变化，chunk 不变
- manifest 不需要 git 追踪（可以加入 .gitignore）

### 替代方案（更简单但粒度粗）

不改上传脚本，改上传策略：
- 构建后只上次 `upload.js` 里指定的目录
- 或者手动在 COS 控制台删掉旧文件再全量上传

**推荐**：方案一（增量上传），一劳永逸。

## 二、已解决：作品栏目中文路径 dev 模式报错

**问题**：`content/works/` 下用中文命名子目录（"文稿"、"设计"），`npm run dev` 访问嵌套页面时报 500：
> Page "/works/[...path]/page" is missing param "/works/[...path]" in "generateStaticParams()"

**原因**：Next.js 16.2.6 (Turbopack) dev 模式下，`generateStaticParams` 对含中文的 catch-all 路由参数匹配失败。英文路径正常。`npm run build` 和生产环境无此问题。

**方案**：子目录统一用英文命名。URL 更干净（`/works/design/` vs `/works/%E8%AE%BE%E8%AE%A1/`），显示可通过 frontmatter `title` 字段保持中文。

**实际执行**：`文稿` → `writing`，`设计` → `design`。dev 模式全线 200。
