## 一、COS 上传耗时分析与增量方案

### 现状

每次 `npm run build` 生成 2177 个文件，上传 COS 耗时 4-7 分钟。但多数时候只改了几篇 `.md` 内容。

### 已完成：增量上传脚本

`upload.js` v2 已实现 SHA256 哈希对比增量上传：

1. 遍历 `out/` 所有文件，计算 SHA256 哈希
2. 从 COS 下载上次的 `upload-manifest.json` 对照
3. 哈希相同的跳过，不同/新增的上传，旧有本地无的从 COS 删除
4. 上传完成后更新 manifest 并存回 COS

没改动内容时重跑上传 → 2177 全跳过，< 1 秒完成。

### 实际问题：不在上传，在构建

增量上传本身是成功的。但实测发现：**改了一篇 .md 内容后 `npm run build`，输出目录里 2177 个文件有 2150 个变了。**

原因：Next.js 的 `_next/static/chunks/` 下的 JS/CSS 文件名包含构建哈希，每次 `npm run build` 都生成不同的哈希值，即使代码和内容都没变。这 2000+ 个 JS/CSS 占了上传时间的 95% 以上。

流程图：

```
改一篇 .md
  → npm run build（全量，30s）
  → out/ 里 2177 个文件几乎全变了 ← 瓶颈在这
  → upload.js 增量对比（秒级）
  → 还是要传 2000+ 个文件（4-7 分钟）← 只是因为构建哈希变了
```

### 探索过的方案（不可行）

**轻量 HTML 替换**：尝试只替换改了内容的 HTML 页面，不动 JS。但 Next.js 页面内容同时嵌在 HTML 和 `self.__next_f.push(...)` 水合脚本里，两者必须一致，否则浏览器报错或评论功能失效。结论：不能只换 HTML。

### 可行方案：上传并行化

当前 `upload.js` 文件逐个串行上传，2177 个文件 = 4-7 分钟。改成 10 个并发上传，时间缩短到 ~30 秒。配合已实现的增量对比（不改动 0 上传），实际体验大幅提升。

**实施**：`upload.js` 加并发控制，每次 10 个文件同时上传。串行 → 并行，零风险，不改构建流程。

## 二、已解决：作品栏目中文路径 dev 模式报错

**问题**：`content/works/` 下用中文命名子目录（"文稿"、"设计"），`npm run dev` 访问嵌套页面时报 500：
> Page "/works/[...path]/page" is missing param "/works/[...path]" in "generateStaticParams()"

**原因**：Next.js 16.2.6 (Turbopack) dev 模式下，`generateStaticParams` 对含中文的 catch-all 路由参数匹配失败。英文路径正常。`npm run build` 和生产环境无此问题。

**方案**：子目录统一用英文命名。URL 更干净（`/works/design/` vs `/works/%E8%AE%BE%E8%AE%A1/`），显示可通过 frontmatter `title` 字段保持中文。

**实际执行**：`文稿` → `writing`，`设计` → `design`。dev 模式全线 200。
