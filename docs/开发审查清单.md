# 开发审查清单

> 每次修改代码后，按此清单逐项检查，确保前端和服务端都正确部署。

---

## 一、核心原则

**每次改动后必须做的事，不假设"应该没问题"。**

### 改动类型与影响范围

| 改了什么 | 需要重新构建前端 | 需要重新部署服务端 | 需要刷新 CDN |
|----------|:---:|:---:|:---:|
| 修改 `.md` 内容文件 | ✅ | ❌ | ✅ |
| 修改前端组件/页面 | ✅ | ❌ | ✅ |
| 修改环境变量 `.env.local` | ✅ 并重启 dev server | ❌ | ✅ |
| 修改服务端路由/逻辑 | ❌ | ✅ | ❌ |
| 修改数据库表结构 | ❌ | ✅ | ❌ |
| 修改 Header/导航 | ✅ | ❌ | ✅ |
| 新增页面 | ✅ | ❌ | ✅ |
| 修改 CORS 配置 | ❌ | ✅ | ❌ |

---

## 二、前端构建检查

### 做任何前端修改之后

1. **构建不能只看"✓ Compiled"，要看是否有 TypeScript 错误**
   ```bash
   npm run build
   ```
   搜索输出中的 `error` 关键字。有 TypeScript 错误不会生成完整的 HTML。

2. **确认新页面文件已生成**
   ```bash
   ls out/新页面路径.html
   ```
   如果新增了路由但 `out/` 里没对应文件，说明构建被 TypeScript 错误中断了。

3. **上传 COS 后确认文件数变化**
   - 新增页面 → 文件数应增加
   - 只改内容 → 文件数应与之前相同

4. **新组件必须使用 `NEXT_PUBLIC_API_URL` 环境变量**
   - ❌ 不要硬编码 `const API = "http://api.xolnxoln.cn"`
   - ✅ 使用 `const API = process.env.NEXT_PUBLIC_API_URL || "https://api.xolnxoln.cn"`
   - 这样本地开发和线上环境都能自动用正确的地址

5. **改 `.env.local` 后必须重启 dev server**
   - 环境变量在 dev server 启动时加载，修改后不重启不会生效

### TypeScript 特别检查

- 新增数组元素时，检查 `as const` 的数组是否也加了对应元素
- 新增状态值时，检查 `useState<"a" | "b">` 的类型联合是否包含了新值
- 新增 props 时，检查所有调用处是否传了新 props

## 三、服务端部署检查

### 做任何服务端修改之后

1. **本地编译通过**
   ```bash
   npm run build  # 或 cd server && npx tsc
   ```

2. **上传 dist/ 到服务器**
   ```bash
   scp -r dist root@193.112.220.113:/srv/blog-api/
   ```

3. **重新构建 Docker 镜像**（不是 restart）
   ```bash
   ssh root@193.112.220.113 "cd /srv/blog-api && docker compose up -d --build"
   ```

4. **验证 API 可用**
   ```bash
   curl https://api.xolnxoln.cn/api/health
   ```

5. **新增路由后验证**
   ```bash
   curl https://api.xolnxoln.cn/api/新路由
   ```

### 如果 SSH 不通

- 用腾讯云 Lighthouse WebShell（浏览器终端）
- `cd /srv/blog-api && docker compose up -d --build`
- dist 文件没传上去的，WebShell 里手动更新

### 服务端修改涉及的文件

| 场景 | 需要更新的文件 |
|------|---------------|
| 新增 API 路由 | `routes/新路由.ts` + `index.ts`（注册路由） |
| 新增数据库表 | `db/schema.ts` + 对应的路由文件 |
| 修改 CORS | `index.ts` |
| 新增管理面板功能 | `routes/admin.ts` + 前端 AdminPanel.tsx |
| 健康检查/中间件 | `index.ts` |

## 五、常见错误速查

| 症状 | 最可能原因 | 快速定位 |
|------|-----------|---------|
| 线上 404（NoSuchKey） | COS 没有这个文件 | `npm run build` 可能被 TS 错误中断 |
| 线上 403 | CDN 没刷新或 COS 权限 | 直连 COS 网站端点测试 |
| 线上看不到新页面 | CDN 缓存 | 刷新 CDN |
| 本地能看线上看不到 | 忘了上传 COS 或 CDN 缓存 | 按第四项全流程检查 |
| `Failed to fetch` | API 地址错误或 CORS | 浏览器直接打开 API URL 看能不能访问 |
| 改代码不生效 | dev server 缓存 | `rm -rf .next && npm run dev` |
| TypeScript 报错但没注意 | 构建"成功"实际有错 | build 输出中搜 `error` |
| 服务端改完不生效 | Docker 没重建 | `docker compose up -d --build`，不是 restart |
| 环境变量改完不生效 | dev server 没重启 | 重启 dev server |
| npm 命令找不到文件 | 工作目录错了 | `cd /Users/xoln/Desktop/编程/博客/site` |
| 某个页面打不开其他正常 | 该页面内容触发渲染死循环 | 查该页 `.md` 有无裸 `#`/`##`/`###` 无文本行 |
| 所有页面同时打不开 | 第一个卡死的页面阻塞了事件循环 | 挨个页面 curl 找第一个 000 的页面 |
