## 一、项目是什么

一个个人博客网站。前端是用 Next.js 写的纯静态页面，托管在腾讯云 COS 上通过 CDN 加速。后端是用 Hono 写的 API 服务，跑在腾讯云 Lighthouse 轻量服务器上，提供评论、留言板、统计、联系表单等功能。

### 第一阶段：方案规划（05-12 ~ 05-13）

**核心问题：我要做个什么？怎么做？**

这个阶段没有写一行代码。产出是 11 份 `.md` 规划文档，放在 `docs/` 目录。每份文档解释一个方面。

**最关键的三个决策**：

1. **先静后动**（博客方案.md）
   - 为什么：个人博客 99% 时间在被人读，不是在被人写。读不需要服务器。HTML 从 CDN 推比任何 SSR 都快。服务器会挂、会欠费、要升级依赖，静态文件只要 COS 桶没删就永远在线。
   - 怎么做：Phase 1 用 Next.js 静态导出（`output: 'export'`）生成纯 HTML/CSS/JS，放 COS。Phase 2 在 Lighthouse 上加一个最小化的 API 服务，只做评论这类天然有状态的功能。
   - *事后验证：这个决策是对的。静态部分零运维，API 挂了不影响页面浏览。*

2. **不用 CMS，直接用 .md 文件**（内容管理与栏目架构.md）
   - 为什么：CMS 多一层管理后台，个人博客用不上。IDE 里写 Markdown 最快，git 给你免费的历史版本。
   - 怎么做：每个栏目是一个文件夹，每篇文章是一个 `.md` 文件，顶部写 YAML 元数据。
   - *事后验证：这个决策也是对的。写 16 篇文章的管理成本和写 100 篇一样，而且备份/迁移就是复制文件夹。*

3. **前端不感知后端是否存在**（我的想法.md）
   - 为什么：Phase 1 没后端的时候页面不能挂。后端挂的时候页面也不能挂。
   - 怎么做：一个 `API_BASE` 环境变量。为空 → 动态功能静默降级（不显示评论框，不报错）。不为空 → 自动激活。
   - *事后验证：这个架构救了无数次。后端调试期间前端完全不受影响。*

**其他文档的作用**：
- **技术选型详解.md**：回答了"为什么选 X 而不是 Y"（Next.js 不是 Astro、MD 不是 MDX、sql.js 不是 better-sqlite3）
- **视觉设计方案.md**：暖白配色（`#fafaf8`，不是纯白）、衬线标题（制造"阅读模式"心理信号）、680px 阅读宽度、1.8 行距、单一强调色蓝
- **技能推荐.md**：Web Design 工具推荐（Figma/Coolors/Lucide/Heroicons/Fontsource）
- **GitHub技能推荐.md**：GitHub Actions/Apps/CLI 的使用清单

### 第二阶段：前端开发（05-14 ~ 05-15）

**核心问题：怎么把 .md 文件变成好看的网页？**

#### 2.1 项目初始化（一行命令）

```bash
npx create-next-app@latest site --typescript --tailwind --app --src-dir --turbopack
```

这给了我们：Next.js 16.2.6、React 19.2.4、Tailwind CSS v4、TypeScript、App Router。

关键配置：`next.config.ts` 中 `output: 'export'`。这一行把 Next.js 从"需要 Node.js 服务器"变成"纯静态文件生成器"。

#### 2.2 内容渲染系统（自写 MarkdownRenderer）

**为什么不用现成的库（react-markdown）？**

零外部依赖意味着：不受库更新影响、完全控制渲染行为、可以加自定义语法（比如 B 站视频嵌入）。

**MarkdownRenderer 的核心逻辑**：

1. 读文件 → 按行分割
2. 遍历每一行，识别类型（标题？代码块？列表？表格？图片？视频？）
3. 对匹配到的行，渲染为对应的 React 元素
4. 对普通行，收集为段落
5. 行内元素（加粗/斜体/链接/代码）用 `parseInline()` 处理

**支持的全部语法**：标题（1-3级）、段落、加粗、斜体、行内代码、链接、无序列表、有序列表、引用、分割线、表格、代码块（带语言标注）、图片（懒加载）、视频（本地 + B站 + YouTube）。

**代码块的 fence 匹配**是一个容易出错的地方。实现的算法：
- 遇到 ` ``` ` 开头的行 → 记录开 fence 用了几个反引号
- 后面只接受同样数量或更多的闭 fence
- 这防止了 ```4 个反引号的代码块被里面 3 个反引号的内容提前关闭```

**自定义扩展语法**：
- `![alt](url)` → 图片，自动 `loading="lazy"`
- `!video[标题](url)` → `<video>` 标签
- `!bilibili[标题](BV号)` → B 站播放器 iframe
- `!youtube[标题](视频ID)` → YouTube 播放器 iframe

#### 2.3 视觉设计实现（CSS 变量 + data-theme）

**配色系统**的心智模型：整个网站只有一个配色方案（不是五颜六色的），用 CSS 变量统一管理。暗色模式不是"换一套配色"，而是"变量指向不同的值"。

```css
:root {
  --bg: #fafaf8;        /* 暖白，不是纯白——久了看纯白像盯着灯泡 */
  --text: #1a1a1a;      /* 非纯黑，略柔 */
  --accent: #2563eb;    /* 蓝色，全站唯一的强调色 */
  --border: #e8e8e4;
}
[data-theme="dark"] {
  --bg: #111110;        /* 暖黑，不是纯黑 */
  --text: #e4e4e0;
  --accent: #60a5fa;    /* 暗色下蓝色要提亮 */
}
```

**字体层级**：标题用衬线（Noto Serif SC）制造"严肃写作"气质；正文用无衬线（Inter + Noto Sans SC）；代码用等宽（JetBrains Mono）。

**装饰元素**的设计原则：有存在感但不抢眼。
- 全站噪点纹理：`body::before` 伪元素，SVG feTurbulence 滤镜，opacity 仅 0.025——几乎看不见但让页面有"纸张感"
- Hero 区域：3 个不同颜色的模糊大圆 + 细网格线，纯 CSS，不加载图片
- 卡片：毛玻璃效果（`backdrop-filter: blur(12px)` + 半透明白底）
- 分割线：渐变消退的横线 + 居中标签，替代默认 `<hr>`

**动画系统**的三个组件：

1. **ScrollReveal**：IntersectionObserver 监听元素是否进入视口。入视口 → 加 `visible` 类 → CSS transition 把 `opacity: 0; translateY(24px)` 过渡到 `opacity: 1; translateY(0)`。**关键设计决策**：默认 `useState(true)` 让内容初始可见，不是先隐藏再显示。
2. **PageTransition**：路由切换时短暂 `opacity-0 translate-y-3`，然后 `opacity-100 translate-y-0`。默认不隐藏，只在切换时做动画。
3. **CSS 工具类**：`.card-hover`（悬浮上移+阴影）、`.icon-bounce`（弹性缩放旋转）、`.btn-ripple`（径向光晕）

#### 2.4 架构重构：从单页到列表+详情（关键转折点）

最初每个栏目是一整页 `.md` 文件——7 个栏目，7 个文件。世界观、人生观、价值观、爱情、亲情、友情全部堆在 `/beliefs` 一个页面里。

问题：内容越多页面越长，读者找不到想看的东西。且每增一篇文章，页面就长一截。

解决方案：拆成**列表页 + 详情页**。

**实现**：
1. 内容重新组织——每篇文章一个独立 `.md` 文件，放对应栏目文件夹
2. 每个 `.md` 顶部加 YAML frontmatter（标题、日期、简介、标签、排序号）
3. `gray-matter` 库解析 frontmatter
4. `getContentList(section)` 读取栏目下所有 `.md` 元数据
5. `getContentItem(section, slug)` 读取单篇完整内容
6. ContentList 和 ContentDetail 两个可复用组件
7. `[section]/[slug]/page.tsx` 动态路由 + `generateStaticParams()`

**排序逻辑**：栏目内所有文章都有 `order` 字段 → 按 order 升序排。任何一篇没有 → 回退到日期降序。

**路由前后对比**：
```
之前：/beliefs         — 一页堆完 6 篇观念
之后：/beliefs         — 列表页，卡片展示 6 条标题+简介
      /beliefs/[slug]  — 每篇点进去是独立详情页
```

#### 2.5 功能页面的实现方式

| 页面 | 实现方式 |
|------|---------|
| `/search` | 自写逐字模糊匹配，构建时生成全站内容 JSON，客户端输入即搜 |
| `/countdown` | `content/countdowns.json` 存日期，`setInterval` 每秒刷新 DOM |
| `/admin` | 密钥登录 → 三个 Tab（评论/留言/消息/统计），PATCH/DELETE 管理操作 |
| 暗色模式 | ThemeScript 内联脚本（渲染前执行）+ ThemeToggle 按钮 + localStorage |

### 第三阶段：后端开发（05-15）

**核心问题：怎么让静态网站有动态功能？**

#### 3.1 服务器环境

买腾讯云 Lighthouse（2C1G，40元/月），选 Docker 应用镜像（预装好 Docker 的开机即用系统）。公网 IP `193.112.220.113`。域名 `api.xolnxoln.cn` A 记录指向这个 IP。

服务器上额外装了 git、Node.js 22、Nginx（`yum install git nginx -y`）。

#### 3.2 API 架构

**为什么选 Hono？** 12KB 的 Node.js 框架，路由即类型（路径参数自动推断类型），内置 CORS/JWT/限流中间件。比 Express 轻得多，比 Fastify 简单。

**为什么选 sql.js？** 纯 JavaScript 的 SQLite。不挑 CPU 架构（Mac ARM 编译的在 Linux x86 上也能跑），不需要编译。Docker 构建 10 秒完成。

**数据库 4 张表**：

```sql
-- 文章评论
CREATE TABLE comments (id, slug, section, author, content, approved DEFAULT 1, created_at);
-- 页面访问记录
CREATE TABLE pageviews (id, path, ip, created_at);
-- 联系表单消息
CREATE TABLE messages (id, name, email, content, read DEFAULT 0, created_at);
-- 留言板
CREATE TABLE guestbook (id, author, content, approved DEFAULT 1, created_at);
```

**15 个 API 接口**（公开 7 个 + 管理 5 个 + 留言板 2 个 + 健康检查 1 个）。

#### 3.3 部署方式

Dockerfile 定义镜像（Node.js Alpine + npm install + 复制 dist/），docker-compose.yml 定义服务（端口映射 + 数据卷挂载 + 自动重启）。Nginx 反代将 `api.xolnxoln.cn` 的请求转发到容器内的 3001 端口。

#### 3.4 前端如何连接后端

`src/lib/api.ts` 是一个统一的 API 调用层。所有函数先检查 `API_BASE` 是否为空：
- 为空（Phase 1）→ 返回空数组/静默降级
- 不为空（Phase 2）→ fetch 调用后端

`.env.local` 设置 `NEXT_PUBLIC_API_URL=https://api.xolnxoln.cn`。改这个变量后重新构建，所有动态功能自动激活。

## 三、14 个问题与解决方法（按发生时间）

### 问题 1：ScrollReveal 导致内容不可见

**什么时候**：2026-05-14，前端视觉开发阶段。

**什么现象**：页面标题能看到，往下的所有内容空白。但 `curl` 抓下来的 HTML 里内容完整存在。用户说："似乎之中有字符，但我就是看不到"。

**排查思路**：
1. 怀疑 PageTransition 的 opacity-0 → 不是
2. HTML 里 `class="scroll-reveal"` 没有 `visible` 类 → 说明 JS 没执行
3. 但标题也在 ScrollReveal 里，标题可见 → 内容的 ScrollReveal 延迟大
4. 标题延迟 0ms，内容延迟 100ms → 这 100ms 内出了问题

**根本原因**：ScrollReveal 默认 `useState(false)`（`opacity: 0`）。MarkdownRenderer 生成的 HTML 在服务器端（Node.js）和客户端（浏览器 React）之间微小差异触发 hydration 异常 → React 卸载重挂组件 → IntersectionObserver 被 `useEffect` 清理函数销毁 → 100ms 的 setTimeout 永远触发不到 → 内容永久透明。

**解决**：改 `useState(true)`，内容默认可见。运行时检查位置：已在视口内 → 保持可见，在视口外 → 才设不可见然后等滚动触发。

**核心教训**：动画组件不应该用隐藏作为初始状态。内容默认可见，动画只是增强。即使 JS 完全崩溃，内容也可读。

### 问题 2：better-sqlite3 跨平台不兼容

**什么时候**：2026-05-15，后端 Docker 部署阶段。

**什么现象**：Docker 容器反复重启。日志：
```
Error: Error loading shared library /app/node_modules/better-sqlite3/build/Release/better_sqlite3.node: Exec format error
code: 'ERR_DLOPEN_FAILED'
```

**排查思路**：`Exec format error` 是 Linux 加载二进制文件时格式不对的标准错误。Mac 是 ARM64，Linux 服务器是 x86_64，CPU 架构不同。better-sqlite3 是 C 原生模块，npm install 时编译为目标平台的二进制。

**尝试的方案**：
- 方案 A：Dockerfile 里装 gcc/python3/make 重新编译。下载耗时 10 分钟。
- 方案 B：换 sql.js——纯 JS，不挑 CPU。npm install 即可。

**最终选择**：方案 B。Docker 构建 10 分钟变 10 秒。个人博客的几十条评论完全感知不到性能差异。

**教训**：能用纯 JS 就别用原生模块。跨平台时 CPU 架构差异是第一大坑。

### 问题 3：Docker 缓存不更新代码

**什么时候**：2026-05-15，后端部署阶段。

**什么现象**：改了代码，`docker compose restart` 之后还是旧的。

**原因**：`restart` 只重启容器，不重建镜像。`COPY dist/ dist/` 在镜像构建时执行。

**解决**：必须 `docker compose up -d --build`。注意 Docker layer 缓存——dist 目录是后来上传的，之前删除过导致 `COPY` 步骤可能用缓存跳过。

**正确流程**：本地编译 → scp 上传 → ssh 执行 `docker compose up -d --build`。

### 问题 4：CORS 跨域

**什么时候**：2026-05-15，前后端连接阶段。

**什么现象**：浏览器 Console：`Failed to fetch`。但 curl 能调 API。

**原因**：后端 CORS 白名单只允许了生产域名，没有 `localhost:3099`。

**解决**：Hono cors 中间件加开发环境 origin。后续又发现 admin 接口 PATCH/DELETE 方法也被 CORS 拦截，又加了 `allowMethods` 和 `allowHeaders`。

---

### 问题 5：HTTPS 环境变量写错

**什么时候**：2026-05-15。

**什么现象**：CORS 修好后仍然 `Failed to fetch`。

**原因**：`.env.local` 写了 `NEXT_PUBLIC_API_URL=https://api.xolnxoln.cn`，但当时服务器没 SSL，443 端口不存在。浏览器 fetch 到 443 直接被拒。

**解决**：改为 `http://`。后续服务器部署 SSL 后改回 `https://`。

**教训**：URL 里的 `http` vs `https` 差一个字母，连接完全不通。

---

### 问题 6：评论存入但查询不到

**什么时候**：2026-05-15。

**什么现象**：curl POST 返回 `{"ok":true}`，但 GET 返回 `[]`，页面上看不到。

**原因**：数据库表是之前用 better-sqlite3 创建的，当时 `approved DEFAULT 0`。改成 sql.js 后 `CREATE TABLE IF NOT EXISTS` 发现表已存在，不会修改表结构。INSERT 没显式写 approved 值，SQLite 用旧默认值 0。查询有 `WHERE approved = 1`。

**解决**：INSERT 显式写 `approved = 1`。

**教训**：`CREATE TABLE IF NOT EXISTS` 只在表不存在时建表。改 schema 不等于改已有表。 

---

### 问题 7：COS 文件上传到子目录

**什么时候**：2026-05-19。

**什么现象**：COS 桶里文件全在 `out/` 前缀下。根路径 `/` 访问不到 `index.html`。

**原因**：`path.join('./out', f)` 返回 `out/index.html`，`file.replace('./out/', '')` 匹配不到（`./out/` ≠ `out/`）。

**解决**：改为 `file.replace('out/', '')`。并清掉所有旧 `out/` 前缀文件。

**教训**：路径处理时 `./` 前缀容易出错，测试上传前先 dry-run 看 key。

---

### 问题 8：网站下载 HTML 不渲染

**什么时候**：2026-05-19，出现两次。

**原因**：上传时没设 `Content-Type` 头，COS 默认 `application/octet-stream`（二进制），浏览器触发下载。

**解决**：上传脚本加 MIME 映射表，`.html` → `text/html`、`.css` → `text/css` 等。

---

### 问题 9：CDN 回源 403（最折腾）

**什么时候**：2026-05-19 ~ 05-20，持续一天半。

**什么现象**：`xolnxoln.cn` 返回 403，响应头 `Server: tencent-cos`，内容是 `<Error><Code>AccessDenied</Code>`。但直接访问 COS 网站端点正常。

**5 次错误判断才定位到根因**：

1. 以为是 DNS → DNS 正确。不是。
2. 以为是桶权限 → 直接访问 200。不是。
3. 以为是回源 HOST → COS 源自动填且不可修改。不是。
4. 以为源站类型错了 → 是 COS 源。不是。
5. 让用户切"自有源"填回源 HOST → 不需要，COS 源有更好的办法。错的。

**关键测试（最终定因）**：
```bash
# COS 标准端点 / → 403
# COS 标准端点 /index.html → 200
# COS 网站端点 / → 200（自动返回 index.html）
```

COS 有两个端点，标准端点是 API 接口没有首页索引，网站端点才有。CDN 配了标准端点。

**解决**：CDN 源站从"默认域名"切为"静态网站"（对网站端点）。

**教训**：诊断先做对比测试——同一个请求换不同端点、换不同路径，马上能缩小范围。

### 问题 13：TypeScript 错误导致页面未生成

## 四、如果重新来一次

1. **第一天就配 CDN**，不要尝试 COS 直连或 Lighthouse 反代。DNS 一开始就 CNAME 到 CDN。
2. **第一时间对比测试**——排查问题先测不同端点（/ vs /index.html，标准 vs 网站），能省一天时间。
3. **上传脚本第一版就设 Content-Type**，不要等到用户看到下载页才改。
4. **写 TypeScript 后跑完整 `npm run build`** 并搜 `error`，不只看 `✓ Compiled`。
5. **不用 `better-sqlite3`**，直接从 sql.js 开始。
6. **动画默认可见**，不需要等到 JS 执行。
7. **CREATE TABLE 时显式设值**，不依赖 DEFAULT。
8. **建《开发审查清单》**，每次改动前对照。

## 六、技术决策速查

| 决策 | 为什么对 | 如果是错的会怎样 |
|------|---------|----------------|
| 纯静态 + COS | 零运维，永远在线 | 每次访问都要服务器渲染，慢且贵 |
| Markdown 不用 CMS | 直接 IDE 写，git 版本管理 | 多了管理后台要维护 |
| 列表+详情 | 无限扩展栏目内容 | 页面越来越长无法阅读 |
| order 排序 | 数字控制文章位置 | 只能按日期排，不好调整 |
| sql.js | 跨平台 10 秒部署 | 每次 Mac→Linux 编译崩 |
| CDN + COS | 全球加速 + 免费 SSL | 反代多一层复杂度 |
| API_BASE 环境变量 | 前后端解耦 | 后端挂了前端白屏 |
| 默认可见的动画 | JS 崩了也能看 | 再次出现"页面空白"bug |
| 开发审查清单 | 每次改动前检查 | 重复踩同样的坑 |

## 七、如果你也想做一个

### 你需要准备的东西

1. **电脑**：能跑 Node.js 的 Mac/Windows
2. **GitHub 账号**：存代码
3. **腾讯云账号**：买 Lighthouse + COS + CDN（要实名认证）
4. **域名**：已备案的最好（备案等 1-2 周）
5. **成本**：域名 ~60 元/年 + Lighthouse ~40 元/月 + COS ~1 元/月

### 你会经历的阶段

1. **写代码**（1-3 天）：跟着教程或者用脚手架快速起项目
2. **调样式**（1-2 天）：配色、字体、动画，改了又改
3. **加功能**（按需）：搜索、评论、留言板...
4. **买服务器**（1 天）：开通、安装环境
5. **部署调试**（3-5 天）：上传、DNS、CDN、HTTPS，这个阶段最折腾
6. **备案**（1-2 周等待）：管局审批，期间网站不能访问
7. **填内容**（持续）：写文章、改关于页、更新倒计时日期

### 你会遇到的坑（提前知道可以避开）

1. DNS 改了要等生效（几分钟到几小时），不要反复改
2. COS 有两个端点，网站托管必须用 cos-website
3. 上传文件没设 Content-Type 会触发下载
4. `npm run build` 的 `✓` 不代表真的成功
5. 从 Mac 上传到 Linux 服务器时，用纯 JS 库不要用原生编译的
6. HSTS/HTTPS 一旦启用，HTTP 就打不开了（浏览器强制跳转）
7. 进学校/公司网可能 SSH 不通，要有备选方案

### 名词解释

| 术语 | 一句话解释 |
|------|-----------|
| COS | 腾讯云的对象存储，可以理解成"高级网盘" |
| CDN | 内容分发网络，把你的网页缓存到全国各地的节点，加速访问 |
| Lighthouse | 腾讯云的轻量服务器，一台在云端 24 小时开机的 Linux 电脑 |
| DNS | 域名解析，把 `xolnxoln.cn` 翻译成 IP 地址 `111.8.3.214` |
| CNAME | DNS 记录类型之一，指向另一个域名而不是 IP |
| A 记录 | DNS 记录类型之一，指向 IP 地址 |
| SSL/HTTPS | 加密访问，浏览器地址栏有小锁 |
| Docker | 容器化工具，把程序和环境打包成一个镜像，到处都能跑 |
| Nginx | 网页服务器，用来做反向代理（把请求转发给后端程序） |
| SQLite | 轻量数据库，单文件存储，不需要安装数据库服务 |
| React | 前端框架，把 UI 拆成组件 |
| Next.js | 基于 React 的框架，支持静态导出 |
| TypeScript | JavaScript 的超集，加了类型检查 |
| Tailwind CSS | 原子化 CSS 框架，用 class 名直接写样式 |
| Markdown | 用纯文本写排版的语法，比 Word 简单 |
