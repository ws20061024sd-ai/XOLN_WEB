## 一、项目概述

用 Next.js 16 + Tailwind CSS v4 构建纯静态个人网站，前端托管在腾讯云 COS + CDN，后端运行在腾讯云 Lighthouse 上提供评论、统计、联系表单等动态功能。

### 最终成果

| 指标 | 数值 |
|------|------|
| 前端静态页面 | 33 个 |
| 栏目数 | 7 个（关于单页 + 5 个列表+详情 + 首页） |
| 工具页面 | 3 个（搜索、倒计时、管理面板） |
| 后端 API 接口 | 12 个 |
| 数据库表 | 3 张（comments、pageviews、messages） |
| 代码行数 | 约 5000 行 |
| 规划文档 | 13 份 |
| 域名 | `https://xolnxoln.cn` |

### 技术栈总览

```
前端：Next.js 16 + TypeScript + Tailwind CSS v4
内容：Markdown + gray-matter frontmatter
渲染：自写 MarkdownRenderer（不依赖外部 Markdown 库）
动画：CSS transition + IntersectionObserver + PageTransition
暗色：CSS 变量 + data-theme + localStorage
后端：Hono + sql.js + Docker + Nginx
数据库：SQLite（单文件，23KB）
部署：COS + CDN（前端），Lighthouse + Docker（后端）
```

## 二、完整实现路径

### 第一阶段：方案规划

**时间线**：2026-05-12 ~ 2026-05-13

**产出**：11 份规划文档

1. **博客方案.md**：制定"先静后动、渐进增强"策略。Phase 1 纯静态 COS 托管，Phase 2 在 Lighthouse 上部署轻量 API。核心理由：个人博客 99% 时间在被人读，静态文件从 CDN 边缘节点推出去比任何 SSR 都快；服务器会挂、会欠费、要升级依赖，静态文件不会

2. **技术选型详解.md**：Next.js 15 → 16、Tailwind v4、Markdown（非 MDX，保持简单）、Hono（12KB API 框架）、SQLite（单文件零运维）

3. **视觉设计方案.md**：暖白底色 `#fafaf8`（非纯白，久读不累）、衬线标题 Noto Serif SC、正文无衬线 Inter、代码 JetBrains Mono、单一强调色蓝色、680px 阅读宽度、1.8 行距

4. **技能推荐.md**：Web Design 工具（Figma/Coolors/Fontsource/Lucide）、GitHub 生态（Actions/Giscus/gh CLI）

5. **内容管理与栏目架构.md**：7 个栏目设计，写作流程（本地 .md → git push → Actions 部署）

6. **我的想法.md**：架构上两个坚持——前端不感知后端是否存在（`API_BASE` 环境变量）、内容归属权在自己手上

关键决策：
- **不用 CMS**：个人博客直接 IDE 写 .md 文件最省事，CMS 多一层管理后台用不上
- **不用 SSR**：静态优先是最佳实践，动态功能本身访问频率低，用独立 API 按需响应即可
- **前端一套代码不分阶段**：Phase 1 和 Phase 2 同一份前端代码，`API_BASE` 为空时动态功能静默降级

### 第二阶段：前端开发

**时间线**：2026-05-14 ~ 2026-05-15

#### 2.1 项目初始化

实现细节：
- `parseInline()` 函数处理行内元素（加粗、斜体、行内代码、链接、图片），用正则匹配找最早出现位置的算法
- 代码块解析改进：匹配开 fence 的反引号数量，只接受同样数量或更多的闭 fence，防止嵌套代码块提前关闭
- 表格自动解析 `|...|` 格式为 `<table>`
- 图片自动 `loading="lazy"` 懒加载
- 自定义语法：`!bilibili[标题](BV号)`、`!youtube[标题](视频ID)`、`!video[标题](URL)`

#### 2.3 视觉设计实现

**字体系统**：
- 标题：`Noto Serif SC` 衬线（制造"阅读模式"心理信号）

- 代码：`JetBrains Mono` 等宽

**装饰元素**：

**动画系统**：

#### 2.4 架构重构：从单页到列表+详情

最初的 7 个栏目每个是一页 `.md` 文件。内容多了会很长，不便利阅读。

重构为列表+详情架构：
- 5 个栏目（观念/作品/喜爱/更新/杂项）拆分为列表页和详情页
- 关于保持单页（个人简介不适合拆分）

**路由变化**：
```

之后：/beliefs（列表页，标题+简介卡片）
      /beliefs/worldview（详情页）
```

**排序逻辑**：
- 所有文章有 `order` → 按 order 升序排列
- 任何一篇无 `order` → 回退到按日期降序
- 用户可精确控制每篇文章的位置

## 三、所有问题与解决方法（按时间顺序）

### 问题 1：ScrollReveal 导致页面内容不可见

**时间**：2026-05-14，视觉开发阶段

**现象**：
页面标题"Markdown 语法测试页"能看到，往下的所有内容全部空白。但是 `curl` 抓下来的 HTML 里内容完整存在。用户描述："似乎之中有字符，但我就是看不到"。

**排查过程**：
1. 首先怀疑 PageTransition 的 opacity-0 → 改为初始 visible，无效
2. 发现 HTML 中 `class="scroll-reveal"` 没有 `visible` 类（opacity: 0）
3. 但标题也在 ScrollReveal 里，标题可见，为什么内容不可见？
4. 标题的 ScrollReveal 延迟为 0，内容的 ScrollReveal 延迟 100ms
5. 推测：100ms 延迟期间发生了 React hydration 异常，组件被卸载重挂

**根本原因**：
`ScrollReveal` 的初始状态 `useState(false)`（opacity: 0）。MarkdownRenderer 产出的复杂 HTML 在服务器和客户端之间存在微小差异（React key 不稳定等），触发 hydration 异常。React 卸载组件重挂，100ms 的 setTimeout 还没触发，IntersectionObserver 被 `useEffect` 的 cleanup 函数断开。内容永久卡在 opacity: 0。

**解决方式**：
改为 `useState(true)`——内容默认可见。JS 运行时检查 `getBoundingClientRect()`：在视口内直接显示，在视口外才先隐藏后等滚动触发。

**教训**：
动画组件不应该用隐藏作为初始状态。内容默认可见，动画只做增强。即使 JS 完全挂了，内容也是可见的。

### 问题 2：better-sqlite3 跨平台不兼容

**时间**：2026-05-15，后端部署阶段

**现象**：
Docker 容器反复重启，日志报错：
```
Error: Error loading shared library /app/node_modules/better-sqlite3/build/Release/better_sqlite3.node: Exec format error
    code: 'ERR_DLOPEN_FAILED'
```

**根本原因**：
`better-sqlite3` 是 C 编译的原生模块。在 Mac（Apple Silicon ARM64）上 `npm install` 编译的二进制是 ARM64 格式，而 Lighthouse 服务器是 x86_64 Linux。二进制格式不兼容。

**尝试的解决方案**：
方案 A：在 Docker 内重新编译。Dockerfile 加 `apk add python3 make g++`，下载编译工具链就要 10 分钟，每次构建都等很久。

方案 B：换用 `sql.js`——纯 JavaScript 实现，不需要编译，不挑 CPU 架构。

**最终选择**：方案 B。Docker 构建从 10 分钟降到 10 秒。个人博客的 SQLite 操作（几十条评论、几百条 PV）完全感知不到和原生模块的性能差异。

**教训**：
能用纯 JS 就别用原生编译的库。跨平台部署时原生模块是最大的坑。Mac ARM → Linux x86 这个问题会一直出现。

### 问题 3：Docker 缓存不更新代码

**时间**：2026-05-15，后端部署阶段

**现象**：
改了服务器代码，`docker compose restart` 之后还是旧的。

**根本原因**：
`restart` 只重启已有容器，不重新构建镜像。`COPY dist/ dist/` 这个步骤在镜像构建时执行，`restart` 不会重新执行。

**解决方式**：
必须用 `docker compose up -d --build` 重新构建镜像。注意：Docker 的 layer 缓存会让 `COPY dist/ dist/` 看起来执行了但实际用了缓存——如果 dist 目录是后来才上传的，之前删除过导致构建失败，缓存状态可能异常。

**正确流程**：
1. 本地 `npx tsc` 编译
2. `scp -r dist root@193.112.220.113:/srv/blog-api/`
3. `ssh root@193.112.220.113 "cd /srv/blog-api && docker compose up -d --build"`

### 问题 4：CORS 跨域

**时间**：2026-05-15，前后端连接阶段

**现象**：
浏览器 Console 报错：`Failed to fetch`，Call Stack 指向 `postComment`。

**根本原因**：
后端 CORS 配置只允许了生产域名 `https://xolnxoln.cn`，没有包含本地开发环境 `http://localhost:3099`。浏览器在 `localhost:3099` 页面发 `fetch` 到 `api.xolnxoln.cn`，跨域被浏览器拦截。

**解决方式**：
Hono 的 cors 中间件添加开发环境 origin：
```ts
cors({
  origin: [
    "https://xolnxoln.cn",
    "https://www.xolnxoln.cn",
    "http://localhost:3099",
    "http://localhost:3000",
  ],
})
```

### 问题 5：HTTPS/HTTP 环境变量不匹配

**时间**：2026-05-15，前后端连接阶段

**现象**：
CORS 修复后仍然 `Failed to fetch`。`curl` 能直接调 API，浏览器不行。

**排查**：
```bash
curl http://api.xolnxoln.cn/api/health  # → 200 ✅
```
浏览器里直接打开 `http://api.xolnxoln.cn/api/health` → 200 ✅

但页面里 fetch 失败。检查 `.env.local`：
```
NEXT_PUBLIC_API_URL=https://api.xolnxoln.cn
```

**根本原因**：
环境变量写的是 `https://`，但服务器没有 SSL 证书，443 端口没有服务。浏览器 fetch 去 443 端口，连接直接被拒。

**解决方式**：
改为 `http://api.xolnxoln.cn`。本地开发阶段用 HTTP。后续给 API 服务器上 SSL 之后再改回 `https://`。

### 问题 6：评论提交成功但页面看不到

**时间**：2026-05-15

**现象**：
`curl` 调用 POST /api/comments 返回 `{"ok":true}`，但 GET 查询返回 `[]`，页面看不到评论。

**根本原因**：
数据库表是在 `better-sqlite3` 时代创建的，当时 `approved INTEGER DEFAULT 0`。虽然后来 `schema.ts` 里改成了 `DEFAULT 1`，但 `CREATE TABLE IF NOT EXISTS` 不会修改已存在的表结构。INSERT 语句没有显式写 `approved` 值，SQLite 使用旧默认值 0。查询使用 `WHERE approved = 1`，评论被过滤。

**解决方式**：
INSERT 时显式写 `approved = 1`：
```sql
INSERT INTO comments (slug, section, author, content, approved) VALUES (?, ?, ?, ?, 1)
```

**教训**：
`CREATE TABLE IF NOT EXISTS` 只在表不存在时建表。表已存在时，修改建表语句不会生效。ALTER TABLE 或显式设值才能真正改变行为。

### 问题 7：COS 文件上传到了子目录

**时间**：2026-05-19，部署上线阶段

**现象**：
COS 桶里列出文件显示所有文件都在 `out/` 前缀下：`out/index.html`、`out/404.html` 等。根路径访问不到 `index.html`。

**根本原因**：
上传脚本中 `outDir = './out'`，`walk` 函数用 `path.join('./out', f)` 返回路径如 `out/index.html`（无 `./` 前缀）。然后用 `file.replace('./out/', '')` 替换，但 `./out/` 不匹配 `out/`，替换失败。

**解决方式**：
改为 `file.replace('out/', '')`。同时用 `cos.deleteMultipleObject` 清掉桶里所有 `out/` 前缀的旧文件，重新上传。

### 问题 8：COS 下载文件而不是显示网页

**时间**：2026-05-19

**现象**：
访问网站时浏览器弹出下载窗口，下载 HTML 文件而不是渲染。

**出现两次**：
第一次：DNS CNAME 指向标准端点，COS 不认识自定义域名，返回错误 XML 下载。
第二次：上传文件没设 Content-Type，浏览器把 HTML 当二进制文件下载。

**根本原因**：
COS 上传时没有设置 `Content-Type` 头部，默认为 `application/octet-stream`（二进制）。浏览器收到后不知道是 HTML，触发下载。

**解决方式**：
上传脚本加入 MIME 类型映射表，根据文件扩展名设置正确的 Content-Type：
```js
const mimeTypes = {
  '.html': 'text/html', '.css': 'text/css', '.js': 'application/javascript',
  '.svg': 'image/svg+xml', '.ico': 'image/x-icon', '.png': 'image/png',
  '.json': 'application/json', '.txt': 'text/plain', '.xml': 'application/xml',
};
```

**时间**：2026-05-19 ~ 2026-05-20

**现象**：
CDN 配置完成后访问 `xolnxoln.cn` 返回 403，响应头 `Server: tencent-cos`，响应体是 COS 的 AccessDenied XML。但直接访问 COS 网站端点返回 200。

**排查过程**（按时间顺序）：

**第 1 次判断**：以为是 DNS 没改对。DNS 解析检查：
```bash
nslookup xolnxoln.cn
# xolnxoln.cn → CNAME → CDN → 111.8.3.214 ✅
```
DNS 正确。不是 DNS 问题。

**第 2 次判断**：以为是 COS 桶权限不对。
```bash
curl "http://xolnxoln-1431302682.cos.ap-guangzhou.myqcloud.com/index.html"
# 200 ✅
```
桶权限正确。不是权限问题。

**第 3 次判断**：以为是回源 HOST 没配。查腾讯云文档，发现 COS 源类型的回源 HOST 是自动填的且不可修改。不是回源 HOST 问题。

**第 4 次判断**：以为是源站类型选错。用户确认是 COS 源，源站地址也正确。不是类型问题。

**第 5 次判断**：让用户切自有源然后手动填回源 HOST。这是错的——COS 源本身支持选静态网站。

**关键测试**（最终定位）：
```bash
# COS 标准端点访问 /
curl "http://xolnxoln-1431302682.cos.ap-guangzhou.myqcloud.com/"
# 403 ❌

# COS 标准端点访问 /index.html
curl "http://xolnxoln-1431302682.cos.ap-guangzhou.myqcloud.com/index.html"
# 200 ✅

# COS 网站端点访问 /
curl "http://xolnxoln-1431302682.cos-website.ap-guangzhou.myqcloud.com/"
# 200 ✅
```

**根本原因**：
COS 有两个访问入口，功能不同：

| 端点 | 地址特征 | 访问 `/` | 访问 `/index.html` |
|------|---------|---------|-------------------|
| 标准端点 | `cos.ap-guangzhou.myqcloud.com` | ❌ 403 | ✅ 200 |
| 网站端点 | `cos-website.ap-guangzhou.myqcloud.com` | ✅ 自动返回 index.html | ✅ 200 |

标准端点是 API 端点，没有首页索引功能。网站端点专门用于静态网站托管，自动把 `/` 映射到 `index.html`。

CDN 的 COS 源配的是"默认域名"（标准端点）。当用户访问 `xolnxoln.cn/` 时，CDN 去 COS 标准端点请求 `/`，COS 返回 403。

**最终解决**：
CDN 源站不要选"默认域名"，选"**静态网站**"（对应 cos-website 端点）。一行配置，改了就行。

**教训**：
- COS 标准端点和网站端点功能完全不同——这是 COS 的核心概念，一开始就该搞清楚
- 诊断要先做对比测试（`/` vs `/index.html`，标准端点 vs 网站端点），对比测试很快能定位
- 不要没看文档就下结论让用户改配置（切自有源那次是错的）

### 问题 10：备案期间域名被拦截

**时间**：2026-05-15 ~ 2026-05-19

**现象**：
```bash
curl http://api.xolnxoln.cn/api/health
# HTTP 302 → Location: https://dnspod.qcloud.com/static/webblock.html?d=api.xolnxoln.cn
```

**根本原因**：
提交 ICP 备案后，域名在审批期间被腾讯云 DNS 强制拦截，任何对该域名的请求都跳转到备案提示页。

**临时解决**：
- 管理面板改用 IP 直连：`http://193.112.220.113`
- 服务器上 Nginx 关机：`systemctl stop nginx`（完全关停网站）
- 备案通过后恢复 DNS 解析 + 重启 Nginx

### 问题 11：HTTPS 证书双重配置的冗余

**时间**：2026-05-19

**背景**：
在 DNS 指向 Lighthouse 做反代期间，用 certbot 给 Lighthouse 上的 Nginx 配了 Let's Encrypt SSL 证书。后来 DNS 切回 CDN 后，CDN 自己提供 SSL 证书。

**结果**：
Lighthouse 上的 Let's Encrypt 证书仍在自动续期（certbot 的定时任务），但实际上不再被用到。CDN 的证书完全接管了 HTTPS。

**影响**：
不影响功能，只是多了一张闲置的证书在自动续期。可以保留（不影响），也可以关掉（但不是必需的）。

### 问题 12：CDN 的 HTTPS 514 错误

**时间**：2026-05-20

**现象**：
HTTP 访问返回 200，HTTPS 返回 514。

**原因**：
CDN 的 HTTPS 配置没开启。虽然证书已部署，但 HTTPS 开关没开。

**解决**：
CDN 控制台 → HTTPS 配置 → 开启 → 开启强制跳转 HTTP → HTTPS。

## 四、走弯路的复盘

### 弯路 1：DNS 反复修改

在 COS 直连、Lighthouse 反代、CDN 加速之间反复调整 DNS。尝试过的方案：
1. CNAME 到 COS 标准端点 → 403
2. CNAME 到 COS 网站端点 → 下载 HTML
3. A 记录到 Lighthouse（Nginx 反代）→ HTTP 200，HTTPS 0（证书在 Lighthouse 上）
4. CNAME 到 CDN → 403、514，反复调试

正确的方案从头就是：**CDN → COS 网站端点**。前三个方案都是中间探索。

### 弯路 2：没先看文档就下判断

让用户切 CDN 源站类型为"自有源"来手动填回源 HOST。实际 COS 源就支持选静态网站，不需要切自有源。

### 弯路 3：没有做隔离对比测试

排查 CDN 403 时，没有第一时间做 `/` 和 `/index.html` 的对比测试，也没有同时测标准端点和网站端点。对比测试能马上缩小范围。

### 弯路 4：上传脚本踩坑多次

- 第一次：路径错误（`./out/` vs `out/`）
- 第二次：路径正确但没有 Content-Type → 下载 HTML
- 第三次：加 Content-Type 但没有清理旧文件 → 残留

每次都要重新上传 295 个文件，耗时且重复。

## 五、最终架构（生产环境）

```
用户浏览器
    │
    ├── https://xolnxoln.cn
    │       ↓ DNS CNAME
    │       CDN（HTTPS 证书由 CDN 提供）
    │       ↓ 回源 HTTP

    │       ↓

    │
    └── https://api.xolnxoln.cn
            ↓ DNS A 记录
            Lighthouse Nginx（80 端口）
            ↓ 反代
            Docker 容器 :3001
            Hono + sql.js
```

## 六、有用的技术决策

| 决策 | 为什么 |
|------|--------|
| 纯 Markdown，不用 CMS | 直接 IDE 写 .md，git 版本管理，不依赖管理后台 |
| 自写 MarkdownRenderer | 零外部依赖，完全控制，支持自定义语法 |
| 列表+详情架构 | 栏目内每篇文章独立一个 .md 文件，无限扩展 |
| frontmatter + order 排序 | 精确控制文章位置，数字调序 |
| sql.js 不是 better-sqlite3 | 纯 JS，不挑 CPU 架构，Docker 构建 10 秒 |
| CDN + COS 不是 Lighthouse 反代 | CDN 自带全球加速 + 免费 SSL，反代多一层复杂度 |
| API_BASE 环境变量 | 前后端解耦，后端不存在前端正常降级 |
| ThemeScript 内联脚本 | 渲染前设 data-theme，防止页面闪烁 |
| 默认可见的动画策略 | 动画是增强，不是入口条件。JS 挂了内容也能看 |
| 内容默认直接显示不审核 | 个人博客流量小，降低交互门槛 |
