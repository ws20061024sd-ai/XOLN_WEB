# 博客技术选型详解

> 实际采用的技术与本文档基本一致。差异：用的是 Next.js 16（不是 15），内容格式是 .md（不是 .mdx），Markdown 渲染是自写组件（不是 react-markdown），搜索是自写逐字匹配（不是 Fuse.js）。

## 前端

### Next.js 15

> 静态生成（SSG）的事实标准

React 元框架，个人博客只用到它两个核心能力：

- **App Router（文件系统路由）**：`src/app/posts/[slug]/page.tsx` 自动映射为 `/posts/hello-world`，无需手写路由配置
- **Static Export**：`output: 'export'` 一行配置，构建产物就是纯 HTML/CSS/JS 文件夹，直接扔 COS

个人博客场景下，Next.js 比 Astro 更重是事实，但 JSX 生态、组件库、社区资源更好。如果追求极致轻量，Astro 是备选——但结合 Phase 2 的 React 组件复用，Next.js 的一致性更好。

关键代码就是两个函数：

```ts
// src/lib/posts.ts
import fs from 'fs'
import path from 'path'
import matter from 'gray-matter'

const postsDir = path.join(process.cwd(), 'content')

// 获取所有文章（构建时运行一次）
export function getAllPosts() {
  const files = walkDir(postsDir)  // 递归遍历所有 .mdx
  return files
    .map(f => {
      const { data, content } = matter(fs.readFileSync(f, 'utf-8'))
      return { ...data, slug: pathToSlug(f), content }
    })
    .filter(p => !p.draft)
    .sort((a, b) => new Date(b.date) - new Date(a.date))
}

// generateStaticParams 告诉 Next.js 构建哪些页面
// posts/[slug]/page.tsx 中：
export function generateStaticParams() {
  return getAllPosts().map(p => ({ slug: p.slug }))
}
```

**适用**：需要 SSG + 将来可能部分 SSR（虽然这里用 API 分离代替了）

### MDX

> Markdown 里写 JSX

普通 Markdown 只能写文本和图片。MDX 让你在文章里直接使用 React 组件：

```mdx
## 一个示例

这是普通 Markdown 段落。

<Callout type="warning">
  这里是**自定义警告框**，可以嵌入任意 React 组件
</Callout>

```js
console.log('代码块照常工作')
```

```

构建时 MDX 编译为 React 组件 → 参与静态生成 → 产出 HTML。

**对比纯 Markdown**：Markdown 需要额外插件才能做代码高亮、表格、脚注；MDX 可以直接用 React 组件接管渲染，更灵活。

### Tailwind CSS v4 + @tailwindcss/typography

> 原子化 CSS + 文章排版

Tailwind 的核心价值在于不用写 CSS 文件。布局、颜色、间距全在 className 里，组件和样式不分离：

```tsx
<article className="prose prose-slate dark:prose-invert max-w-none">
  {mdxContent}
</article>
```

`@tailwindcss/typography` 的 `prose` 类自动处理文章的排版——标题层级、段落间距、列表样式、引用块、代码块。不需要手动给文章内容里的每个元素写样式。

Tailwind v4 相比 v3 的改进：不再需要 `tailwind.config.ts`，全部在 CSS 里配置，构建更快。

### Fuse.js

> 客户端全文搜索，零后端

构建时生成一个 JSON 索引文件（文章标题 + 摘要 + 标签），搜索时加载这个 JSON，在浏览器里跑 Fuse.js：

```
搜索流程：
用户输入关键词 → Fuse.js 在 JSON 索引中模糊匹配 → 渲染结果列表
```

JSON 索引一般几十 KB，浏览器加载毫秒级。个人博客文章量（几十到几百篇）完全够用。

为什么不用 Algolia/DocSearch？不需要第三方服务，不需要 API 调用，离线也能搜。

### rehype-pretty-code

> 代码块语法高亮

基于 Shiki（VS Code 同款高亮引擎），支持：

- 行号、行高亮
- 文件名标题栏
- 暗色/亮色双主题
- 复制按钮通过 CSS + 少量 JS 实现

```tsx
// MDX 组件定制示例
// 在 mdx-components.tsx 中覆盖 pre 和 code：
export function useMDXComponents(components) {
  return {
    pre: CodeBlock,   // 自定义代码块组件（包装 pretty-code 输出）
    img: ProseImage,  // 自定义图片（懒加载 + 灯箱占位）
    ...components,
  }
}
```

## 后端（Phase 2）

### Hono

> 轻量 Node.js API 框架，路由即类型

Express 太重，Fastify 有点过度设计。Hono 的几个优势很适合个人博客 API：

- **体积极小**：12KB minified，冷启动 < 50ms
- **路由即类型**：路径参数自动推断类型，`/comments/:slug` 里 `slug` 直接有类型
- **中间件齐全**：CORS、JWT 认证、限流都内置
- **可部署到边缘**：如果以后想迁移到 Cloudflare Workers 或腾讯云 SCF，代码不用改

一个评论接口长这样：

```ts
// server/src/index.ts
import { Hono } from 'hono'
import { cors } from 'hono/cors'
import { zValidator } from '@hono/zod-validator'
import { z } from 'zod'

const app = new Hono()

app.use('/api/*', cors({ origin: 'https://yourdomain.com' }))

// GET /api/comments/:slug
app.get('/api/comments/:slug', async (c) => {
  const slug = c.req.param('slug')
  const comments = await db.query(
    'SELECT * FROM comments WHERE slug = ? AND approved = 1 ORDER BY created_at DESC',
    [slug]
  )
  return c.json(comments)
})

// POST /api/comments
app.post('/api/comments',
  zValidator('json', z.object({
    slug: z.string(),
    author: z.string().min(1).max(50),
    content: z.string().min(1).max(5000),
  })),
  async (c) => {
    const body = c.req.valid('json')
    await db.run(
      'INSERT INTO comments (slug, author, content) VALUES (?, ?, ?)',
      [body.slug, body.author, body.content]
    )
    return c.json({ ok: true }, 201)
  }
)

export default app
```

不到 30 行实现评论 API，包括参数校验。

### SQLite + better-sqlite3

> 零配置、单文件数据库

为什么不用 MySQL/PostgreSQL？

- SQLite 不需要安装服务端，一个 `blog.db` 文件就是整个数据库
- 读写性能在个人博客体量下完全过剩（SQLite 每秒能处理几万次查询）
- 备份 = 复制一个文件
- Lighthouse 上不需要额外内存开销跑 MySQL 进程

`better-sqlite3` 是 Node.js 下最快的 SQLite 驱动，同步 API，代码简洁：

```ts
import Database from 'better-sqlite3'

const db = new Database('data/blog.db')

// 建表（应用启动时执行一次）
db.exec(`
  CREATE TABLE IF NOT EXISTS comments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    slug TEXT NOT NULL,
    author TEXT NOT NULL,
    content TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
  )
`)
```

如果以后想写 SQL 太麻烦，可以加一层 Drizzle ORM——但它本质也是生成 SQL，个人博客 CRUD 用原生 SQL 更透明。

### Lighthouse 部署

```
Docker Compose 一键拉起：

blog-api/
├── docker-compose.yml
├── Dockerfile
├── data/                  # SQLite 数据挂载到宿主机
└── src/

docker-compose.yml:
  services:
    api:
      build: .
      ports: ["3001:3001"]
      volumes: ["./data:/app/data"]
      restart: always
```

Nginx 反代配置：

```nginx
server {
    listen 443 ssl;
    server_name api.yourdomain.com;

    ssl_certificate     /etc/ssl/certs/yourdomain.pem;
    ssl_certificate_key /etc/ssl/keys/yourdomain.key;

    location / {
        proxy_pass http://127.0.0.1:3001;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

## 运维/工具

### COSCMD

腾讯云官方 CLI 工具，上传静态文件到 COS：

```bash
coscmd upload -r out/ / --delete --ignore "*.map"
# -r: 递归上传
# --delete: 删除 COS 上多余文件（保持同步）
# --ignore: 不上传 sourcemap
```

配合 npm script：`"deploy": "next build && coscmd upload -r out/ /"`

### GitHub Actions（可选）

```yaml
name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci && npm run build
      - run: npm run deploy          # COSCMD 上传
        env:
          COS_SECRET_ID: ${{ secrets.COS_SECRET_ID }}
          COS_SECRET_KEY: ${{ secrets.COS_SECRET_KEY }}
```

## 不推荐或不用的

| 技术 | 原因 |
|------|------|
| WordPress / Typecho | 太重，需要 PHP + MySQL，和静态博客理念冲突 |
| MongoDB | 个人博客不需要 NoSQL，SQLite 更轻 |
| Redis | 个人博客体量不需要缓存层 |
| Prisma ORM | 太重，冷启动慢，SQLite 场景下 Drizzle 或原生 SQL 更合适 |
| Kubernetes | 单服务器不需要编排，Docker Compose 足够 |
| WebSocket | 博客不需要实时双向通信 |

---

## 总结：最小可行技术栈

```
构建时：  MDX 文件 → Next.js (export) → out/ 目录 → COSCMD → 腾讯云 COS
运行时：  浏览器 ← COS + CDN ← 用户
         浏览器 ← api.yourdomain.com ← Nginx ← Hono ← SQLite（Phase 2）
```

核心就记住三个文件：`.mdx` 写文章，`page.tsx` 定义页面，`posts.ts` 读取文章列表。其余组件和样式围绕它们服务。
