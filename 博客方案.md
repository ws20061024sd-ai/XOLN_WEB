# 个人博客网站方案

## 总体策略：先静后动，渐进增强

```
Phase 1（纯静态）                Phase 2（动静分离）
┌─────────────┐                 ┌─────────────┐
│   COS 对象   │                 │   COS 对象   │
│   存储       │                 │   存储       │
│  (静态前端)  │                 │  (静态前端)  │
└─────────────┘                 └──────┬──────┘
       │                               │  fetch / fetch
       ▼                               ▼
  用户浏览器                        用户浏览器 ────────▶ ┌──────────────┐
                                                        │  Lighthouse   │
                                                        │  轻量服务器    │
                                                        │  (API 后端)   │
                                                        └──────────────┘
```

- **Phase 1**：构建纯静态文件，上传到腾讯云 COS，所有功能走客户端（搜索、RSS 等）
- **Phase 2**：在 Lighthouse 上部署轻量 API，前端通过 `fetch` 请求后端，按需激活评论、统计等功能

## 一、技术选型

### Phase 1：纯静态生成

| 层级 | 技术 | 理由 |
|------|------|------|
| 构建工具 | Next.js 15 `output: 'export'` | 开发体验好，静态导出直接是一堆 HTML/CSS/JS，扔到 COS 就能跑 |
| 内容 | MDX (Markdown + JSX) | 本地 .mdx 文件写作，构建时编译为静态 HTML |
| 样式 | Tailwind CSS v4 + `@tailwindcss/typography` | 原子化 CSS + 文章排版 prose 类 |
| 部署 | 腾讯云 COS 静态网站托管 | 已就绪，配合 CDN 加速 |
| 上传 | COSCMD / `cos-nodejs-sdk-v5` | 构建后一键上传到 COS 桶 |
| 搜索 | Fuse.js（客户端） | 构建时生成 JSON 索引，浏览器端搜索，零后端 |
| RSS | `feed` 包构建时生成 XML | 随静态文件一起上传 |
| 评论 | 暂无（Phase 1） | 可先用 Giscus 过渡，Phase 2 切换到自建 |

### Phase 2：轻量化后端

| 层级 | 技术 | 理由 |
|------|------|------|
| 服务器 | 腾讯云 Lighthouse | 轻量应用服务器，2C2G 几十块/月，跑 Node 足够 |
| API 框架 | Hono 或 Fastify | 轻量、快、TypeScript 原生支持 |
| 数据库 | SQLite (Turso / `better-sqlite3`) | 无需安装数据库服务，单文件存储，Lighthouse 上直接跑 |
| 评论 | 自建 API + 审核 | 脱离第三方依赖 |
| 统计 | 自建埋点 API | 简单的 PV/UV 统计 |
| 认证 | GitHub OAuth（评论用） | 轻量身份验证，无需自建用户系统 |
| 部署 | Docker / PM2 | 容器化或进程守护 |

### 为什么不用 SSR（全程服务端渲染）

| 对比 | 纯静态 + COS | 全栈 SSR（Vercel / Lighthouse） |
|------|-------------|-------------------------------|
| 首屏速度 | 极快（CDN 边缘节点） | 取决于服务器位置 |
| 成本 | COS 几乎免费 | 需要持续运行的服务器 |
| 维护 | 零运维 | 需要维护 Node 进程 |
| 动态能力 | 无（Phase 2 补上） | 天然支持 |
| 构建频率 | 写完文章本地 build + push | 可以按需 ISR |

**结论**：对于个人博客，静态优先是最佳实践。动态功能本身访问频率低，用独立 API 按需响应即可，不值得让整个站点 SSR。

## 四、构建与部署流程

### Phase 1：本地构建 → COS

```
本地写作                   构建                      部署
┌──────────┐    ┌──────────────────┐    ┌──────────────────┐
│ 写 .mdx  │    │ next build       │    │ COSCMD upload    │
│ 文章     │───▶│ (output: export) │───▶│ out/ → COS 桶    │
│          │    │ → out/ 目录      │    │                  │
└──────────┘    └──────────────────┘    └──────────────────┘
```

**next.config.ts 关键配置：**
```ts
const nextConfig = {
  output: 'export',
  images: { unoptimized: true },  // COS 不支持 next/image 优化
}
```

**上传脚本（scripts/upload.mjs）：**
```js
// 使用 cos-nodejs-sdk-v5 将 out/ 上传到指定 COS 桶
// 设置 Cache-Control、Content-Type
// 可配合 CDN 刷新
```

**后续可升级为 CI：** GitHub Actions 监听 push → 构建 → COSCMD 上传 → 刷新 CDN。

### Phase 2：Lighthouse 部署 API

```
Lighthouse 服务器
├── /srv/blog-api/           # API 服务（Hono/Fastify）
│   ├── src/
│   ├── data/
│   │   └── blog.db          # SQLite 数据库文件
│   └── docker-compose.yml
├── Nginx（反向代理 + HTTPS）
│   └── api.yourdomain.com → localhost:3001
└── CORS 白名单：yourdomain.com
```

**API 域名规划：**
- 博客前端：`yourdomain.com`（COS + CDN）
- API 端点：`api.yourdomain.com`（Lighthouse）
- CORS 配置：仅允许 `yourdomain.com`

**Lighthouse 选型参考（腾讯云）：**
| 套餐 | 配置 | 月费 | 适合 |
|------|------|------|------|
| 入门版 | 2C1G, 40GB SSD | ~40 元 | 纯 API，绰绰有余 |
| 通用版 | 2C2G, 60GB SSD | ~65 元 | API + 跑更多服务 |
| 进阶版 | 2C4G, 80GB SSD | ~90 元 | 有余量跑 Docker |

> SQLite 单文件数据库对个人博客 API 完全够用，每秒几百次查询毫无压力。后续想升级可以迁移到 MySQL（Lighthouse 也支持一键安装）。

## 五、Phase 1→2 迁移策略

关键设计原则：**前端代码不感知后端是否存在**。

```ts
// src/lib/api.ts —— 统一的前端 API 调用层

const API_BASE = process.env.NEXT_PUBLIC_API_URL || ''  // Phase 1 为空字符串

export async function fetchComments(slug: string) {
  if (!API_BASE) return []  // Phase 1：返回空，评论功能静默降级
  const res = await fetch(`${API_BASE}/comments/${slug}`)
  return res.json()
}
```

这样做的效果：
- Phase 1 构建时不依赖后端，浏览时评论等动态模块显示"暂无评论，敬请期待"
- Phase 2 设置 `NEXT_PUBLIC_API_URL=https://api.yourdomain.com` 后重新构建，动态功能自动激活
- 前端代码始终只有一套，不分支

**平滑切换清单：**

| 步骤 | 操作 |
|------|------|
| 1 | Lighthouse 服务器就绪，部署 API 服务 |
| 2 | 配置 Nginx + SSL（Let's Encrypt / 腾讯云免费证书） |
| 3 | 前端设置 `NEXT_PUBLIC_API_URL`，重新 build + 上传 COS |
| 4 | 动态功能上线 |

## 八、可选增强

- **图片优化**：Phase 1 可先用 `sharp` 脚本本地生成缩略图和多尺寸 WebP。Phase 2 可在 Lighthouse 上做图片上传 + 处理 API。
- **OG Image**：使用 `satori`（运行在边缘，不依赖浏览器）构建时生成社交分享卡片。
- **CI/CD**：GitHub Actions → `next build` → COSCMD upload → CDN purge，全程自动化。
- **PWA**：`next-pwa` + Service Worker，让博客离线可访问（纯静态博客做 PWA 体验极好）。
- **国际化**：`next-intl` 支持中英双语路由。
